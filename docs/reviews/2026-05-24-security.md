## Verdict

No critical issues. 1 High (vulnerable transitive `aiohttp` in Docker images), 3 Medium, 4 Low. ## High

### H1 — Docker image may ship vulnerable `aiohttp` (CWE-1395 / vulnerable dependency)

**File:** `/Users/igor.khomiakov/Code/usdt_monitor_bot/Dockerfile:17-18`

**Threat model:** The container is the deployed artefact. `pip-audit` against the resolved env reports `aiohttp 3.13.2` carries 18 CVEs (e.g. CVE-2026-34515..34520, CVE-2025-69223..69230); fixed in `3.13.4`. `uv.lock` already resolves `aiohttp==3.13.5`, but the Dockerfile runs `uv pip install --system .` without copying or referencing `uv.lock`, so each image build re-resolves transitives and is non-reproducible. Any attacker who can reach Telegram's outbound HTTP path (or any of the bot's outbound endpoints) could trigger aiohttp parsing bugs server-side. Also affects the CI `security.yml` job which audits the runner's freshly-resolved env, not what ships.

**Fix:** `COPY uv.lock .` + `uv sync --frozen --system` (or `uv pip install --system --requirement uv.lock`). Same change feeds the `security.yml` audit. **→ [#215](https://github.com/hostmaster/usdt_monitor_bot/pull/215) merged: adds `uv.lock` to build context, uses `--frozen` install, pins `security.yml` to audit the locked env.**

## Medium

### M1 — No pinned base image digest, no multi-stage build (CWE-1357 / supply chain)

**File:** `/Users/igor.khomiakov/Code/usdt_monitor_bot/Dockerfile:1, 7`

**Threat:** `FROM python:3.14-alpine` and `COPY --from=ghcr.io/astral-sh/uv:latest /uv ...` are floating tags. A compromised registry tag can inject a backdoor at next build. `uv:latest` is especially dangerous — it'll change silently.

**Fix:** Pin by SHA256 digest (`python:3.14-alpine@sha256:...`, `ghcr.io/astral-sh/uv:0.x.y@sha256:...`). Renovate already handles digest pins. Drop `uv` from final image with a multi-stage `FROM ... AS builder` to shrink attack surface. **Partial → [#215](https://github.com/hostmaster/usdt_monitor_bot/pull/215) merged: pins `python:3.14-alpine` → `python:3.14.3-alpine3.22` and `uv:latest` → `uv:0.11` (version tags, not SHA digests). SHA digest pinning is the remaining step — Renovate can manage this.**

### M2 — GitHub Actions pinned by major tag, not SHA (CWE-829)

**Files:** `.github/workflows/docker.yml`, `security.yml`, `test.yml`, `lint.yml`, `hadolint.yml`

**Threat:** All third-party actions use floating tags (`actions/checkout@v6`, `docker/login-action@v4`, `astral-sh/setup-uv@v8.1.0`, `hadolint/hadolint-action@v3.3.0`, `astral-sh/ruff-action@v4.0.0`, `docker/build-push-action@v7`). A compromised upstream tag (cf. tj-actions incidents) instantly executes attacker code on a runner that has `packages: write` to `ghcr.io/hostmaster/usdt_monitor_bot` plus `GITHUB_TOKEN`. Workflow runs on `push: main` only, but the security workflow runs on `pull_request` (from forks defaults to read-only, so lower risk for that one specifically).

**Fix:** Pin all actions to full commit SHAs and add a Dependabot/Renovate rule for `github-actions`. Repo has no `.github/dependabot.yml` and the existing `renovate.json` doesn't restrict to digests for actions.

### M3 — No secret-scanning pre-commit / baseline

**Threat:** No `.pre-commit-config.yaml`, no `.secrets.baseline`, no `gitleaks`/`detect-secrets`. Operators could `git commit` an `.env` despite `.gitignore` (e.g. forced add). History was spot-checked with `git log -S "BOT_TOKEN="` — clean today, but no guardrail prevents a slip.

**Fix:** Add `detect-secrets` or `gitleaks` as a pre-commit hook + CI job.

## Low

### L1 — `BLOCKSCOUT_BASE_URL` not scheme/host-validated (CWE-918 / SSRF, operator-trust model)

**Files:** `usdt_monitor_bot/config.py:253-255`, `blockscout.py:63,185`

**Threat:** An operator (not attacker) could set `BLOCKSCOUT_BASE_URL=http://169.254.169.254/...` and the process would happily probe the IMDS or other internal endpoints every 60s. Not exploitable by Telegram users; only flagging because the variable is documented as user-configurable in `CLAUDE.md`. Same applies to `ETHERSCAN_BASE_URL`.

**Fix:** Validate scheme is `https` (with `http` allowed only via explicit opt-in), and reject RFC1918 / link-local hosts.

### L2 — Attribute injection via `tx_hash` in `hlink` href (CWE-79, theoretical)

**File:** `usdt_monitor_bot/notifier.py:126, 143`

**Threat:** `hlink('View', f'{explorer_url}/tx/{tx_hash}')` — aiogram's `html_decoration.link` does not HTML-escape the `href` attribute value (see `.venv/.../aiogram/utils/text_decorations.py:234`). `tx_hash` is only validated `startswith("0x")` (notifier.py:63), not hex-only/length. A hostile upstream API (Etherscan/Blockscout/Moralis) returning `0x"><script>...` could break out of the attribute. Telegram's own HTML parser is strict (rejects messages on parse fail), so practical impact is a failed `send_message` rather than code execution.

**Fix:** Enforce `^0x[a-fA-F0-9]{64}$` for `tx_hash` in `_format_token_message`, same way as Eth addresses; same treatment for `address_to_show` (currently only checks `startswith("0x")`).

### L3 — `USDT_CONTRACT_ADDRESS` / `USDC_CONTRACT_ADDRESS` flow unvalidated into HTML href

**File:** `usdt_monitor_bot/config.py:346, 354, 363, 371`

**Threat:** Operator-set env var interpolated into `explorer_url=f"https://etherscan.io/token/{final_contract}"`; that string is later concatenated into a Telegram `href`. Operator-trust, not user-controlled, but no validation. Same fix as L2 — validate the address regex once on load.

### L4 — `.dockerignore` is permissive

**File:** `/Users/igor.khomiakov/Code/usdt_monitor_bot/.dockerignore`

**Threat:** `data/`, `application.log`, `debug.log`, `baseline.json`, `docs/`, `tests/` are not excluded. Practical exposure is nil because the Dockerfile only `COPY pyproject.toml .` + `COPY usdt_monitor_bot/ usdt_monitor_bot/` (not `COPY . .`), but the safety net is one line away from a regression.

**Fix:** Add `data/`, `*.log`, `tests/`, `docs/`, `.github/`, `*.md`, `Taskfile.yml` to `.dockerignore`.

## Verified clean

- **Secret handling:** `TELEGRAM_BOT_TOKEN`, `ETHERSCAN_API_KEY`, `BLOCKSCOUT_API_KEY`, `MORALIS_API_KEY` are only read from env in `config.py`; none appear in any `logging.*` call; tracebacks logged via `exc_info=True` do not re-emit query params; `.env` not in git history (`git ls-files .env` → not tracked), `.gitignore` covers `.env`, `*.log`.
- **SQL injection:** Every `cursor.execute` uses `?` placeholders. The two f-string SQL spots (`database.py:154-155, 578`) interpolate only internal constants / placeholder-count strings — no user data.
- **Authn / multi-tenancy:** All `/list`, `/remove`, `/spam` queries scope by `user_id = ?` (handlers.py:121,154,183; database.py:293,301,675,727). `/spam` is not admin-only — it's per-user and only returns spam for that user's wallets.
- **All handlers gated** `F.chat.type == "private"` — bot won't process group chat injection vectors.
- **Address validation:** strict `^0x[a-fA-F0-9]{40}$` fullmatch at command intake (`handlers.py:24,32`).
- **HTML escaping:** aiogram `hbold`/`hcode` HTML-escape user-supplied `full_name`/`username` (see `markdown.py:40,84`, `text_decorations.py:264-265`).
- **No `eval`/`exec`/`shell=True`/`os.system`/`verify=False`/`ssl=False`** anywhere in `usdt_monitor_bot/` or `tests/`.
- **Container hardening:** Dockerfile creates `appuser:appgroup` and uses `USER appuser` (Dockerfile:22-27), `PYTHONDONTWRITEBYTECODE=1`, explicit `STOPSIGNAL`.
- **GitHub Actions permissions:** `contents: read` on lint/security/test/hadolint; `contents: read, packages: write` on docker.yml only — least privilege observed.
- **CI security tooling:** `bandit` + `pip-audit` already run in `security.yml`; baseline mechanism guards against silently-introduced bandit findings.
- **Telegram-side spoofing:** uses `message.from_user.id` (server-asserted by Telegram); no `from` impersonation possible from bot's vantage point.
